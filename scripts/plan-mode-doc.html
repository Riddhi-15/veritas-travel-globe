5. Stage 1 — Trip Builder

Stage 1 of 5 — Select Destinations & Configure

What It Contains

  • Globe enters selection mode — clicking a country adds it to the trip instead of opening the info panel
  • Each selected country shows a numbered pin on the globe (①, ②, ③…)
  • Left sidebar lists selected destinations in order
  • Per destination controls:
    • Optional city search to narrow down to a specific city
    • Days slider: 1–14 days (default: 3 days)
    • Tier selector: Budget Mid-range Luxury
    • Remove button ✕
  • Destinations can be reordered (up/down arrows)
  • "Plan My Trip →" button activates when at least 1 destination is added
  • Trip data persisted in localStorage — survives page refresh

Existing Code Reused

  • Globe click handler — modified to call addTripDestination()
  • FALLBACK data — country names and iso2 codes
  • Zustand store — add tripDestinations[] array and planMode boolean
  • City search bar — same Nominatim utility already built

6. Stage 2 — Best Month Finder

Stage 2 of 5 — Find the Best Travel Month for All Destinations

What It Contains

  • 12-month grid with one row per selected destination
  • Each cell colour-coded: Peak ● Okay ░ Avoid ✗
  • Months where no destination is red are highlighted with a glow border
  • Recommended month displayed prominently: "Best month for all: October"
  • If no perfect month: "No single ideal month — closest options: Sep, Nov"
  • Replaces the News section — news is discovery, planning needs decisions

Algorithm (Zero New APIs)

function findBestMonths(destinations) {
  const months = ['jan','feb','mar','apr','may','jun',
                  'jul','aug','sep','oct','nov','dec']
  return months.filter(m =>
    destinations.every(d => String(d[`bt_${m}`]) !== '3')
  )
}

Runs entirely on the bt_jan through bt_dec fields already present in fallback.js and cities.js. No API call required.

9B. Per-Destination Block Structure

Each destination gets a dedicated block containing:

  • Country / city name, date range (Day X to Day Y)
  • Visa requirement summary and advisory level
  • Daily cost at selected tier in local currency + INR
  • Top 5 places distributed across the days
  • Hidden gem callouts marked with ✦

9C. Day-by-Day Schedule Generation

Pacing Rules:

  • Day 1 of each destination = Arrival + orientation (no places assigned)
  • Last day of each destination = Departure / transit (no places assigned)
  • Middle days distribute the 5 curated places evenly
  • If only 1–2 days total: Day 1 = arrive + 2 places, Day 2 = 3 places + depart
  • Transit days auto-inserted between destinations

9E. Generation Logic

function buildItinerary(tripDestinations) {
  let currentDay = 1
  const sections = []

  for (const dest of tripDestinations) {
    const places   = dest.places        // already sorted by rank (1–5)
    const daysHere = dest.days
    // Day 1 = arrival, last day = departure, middle = place visits
    const middleDays = Math.max(1, daysHere - 2)
    const placesPerDay = Math.ceil(places.length / middleDays)

    sections.push({
      dest,
      startDay: currentDay,
      endDay: currentDay + daysHere - 1,
      places,
      placesPerDay,
    })

    currentDay += daysHere + 1  // +1 for transit day between destinations
  }

  return sections
}

No AI required. No external API. Pure JavaScript algorithm running on existing data.

10. Export Options

Option How It Works Code Needed Library
Copy as Text Copies plain text itinerary to clipboard 2 lines — navigator.clipboard.writeText(text) None
Save as PDF window.print() with @media print CSS that hides globe, shows only itinerary 5 lines CSS + 1 line JS None — built into every browser
Download .txt Creates a Blob and triggers anchor download 4 lines JS None

Decision: Do NOT use jsPDF or pdfmake — they add 300–500KB to the bundle and produce worse results than window.print(). The browser's built-in print-to-PDF is free, zero dependencies, and produces clean output.

Idea Why to Avoid
User login / accounts Requires backend, database, authentication library — completely different project scope
"Book now" / flight search Skyscanner and Kayak APIs are paid and require business accounts

14. Viva Talking Points

On Plan Mode — What to Say

"The synopsis described Plan Mode as future architectural groundwork. We went beyond that and actually implemented it. The itinerary generator uses no AI and no external APIs — it is a pure algorithm running on the same data already loaded for Browse Mode. The entire trip plan — from best month calculation to day-by-day schedule to budget breakdown to visa checklist — is generated client-side in milliseconds with zero server cost."

On the Best Month Finder — What to Say

"This is the feature that doesn't exist anywhere else. Google Flights tells you prices. Skyscanner tells you prices. Nothing shows you a visual month-by-month overlap grid for multiple destinations simultaneously using seasonal travel data. We built this entirely from the bt_jan through bt_dec fields already in our data — no new API, no new data, just a filter algorithm on existing fields."

On No AI — What to Say

"We deliberately avoided AI generation for the itinerary. AI APIs cost money per request, can give incorrect travel information, and require exposing API keys. Our approach — distributing curated, manually verified places across days using a simple pacing algorithm — is more reliable, faster, completely free, and works offline."

Key Questions & Answers

Question Answer
Why no backend? Google Sheets acts as a live CMS via the gviz CSV endpoint. No auth, no server, no cost. Content updates happen by editing a spreadsheet.
What if an API fails? Every data request has a 3-tier fallback: check sessionStorage cache → fetch from Sheets → use bundled fallback.js. App works fully offline.
How does the itinerary know how many days? The user sets days per destination in Stage 1 (the Trip Builder). The algorithm distributes the 5 curated places across the middle days, reserving Day 1 for arrival and the last day for departure.
Is this original work? Yes — the multi-destination best-month overlap algorithm, the 3-tier data fallback architecture, the Sheets-as-backend pattern, and the itinerary generator are all original design and implementation decisions made by our team.
